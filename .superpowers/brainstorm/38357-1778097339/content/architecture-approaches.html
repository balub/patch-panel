How should metrics flow through the system?

Three architectural approaches — pick the one that fits your goals

A

Pull Chain (Simplest)

Each agent exposes a local HTTP API. The central hub periodically pulls from all agents. The extension polls the hub.

[Machine A Agent :8080] ←poll─┐
[Machine B Agent :8080] ←poll─┤ Central Hub
[Machine C Agent :8080] ←poll─┘ ↑
                       poll every N sec
                       Chrome Extension

Pros

  • Simple to implement
  • Agents need no outbound config
  • Easy to debug

Cons

  • Hub must reach all agents (same network)
  • Not great for agents behind NAT
B

Push + SSE ✦ Recommended

Agents push metrics to the hub on a configurable interval. Hub streams updates to the Chrome extension via Server-Sent Events (SSE).

[Machine A] ──push──┐
[Machine B] ──push──┤ Central Hub ──SSE stream──→ Extension
[Machine C] ──push──┘ (ring buffer, 1hr history)
                configurable push interval

Pros

  • Works across NAT (agents initiate)
  • SSE is browser-native — no WebSocket complexity
  • Near-real-time feel in extension

Cons

  • Hub must be reachable by agents
  • Extension holds open SSE connection
C

WebSocket Mesh

Agents and the extension all maintain persistent WebSocket connections to the hub. Hub fans out metrics to all connected clients in real time.

[Machine A] ←──WS──┐
[Machine B] ←──WS──┤ Central Hub ←──WS──→ Extension
[Machine C] ←──WS──┘ (can request on-demand updates)
               bi-directional, true real-time

Pros

  • Truly real-time, sub-second updates
  • Hub can push commands to agents

Cons

  • More complex to implement
  • Reconnect logic needed on both sides
