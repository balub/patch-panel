Revised Architecture: Follow the Homarr Standard

Your extension talks to Dash. / Glances — just like Homarr does. Pick where the aggregation lives.

A

Extension-as-Aggregator ✦ Recommended

No server needed. The extension's background service worker polls Dash./Glances directly. Readings cached in chrome.storage.local for sparklines.

┌─────────────────────────────────────────┐
│ patch-panel Extension (background SW) │
│ │
│ poll /load/cpu, /load/gpu, /info... │
│ ↓ ↓ ↓ │
│ [Dash. :3001] [Glances :61208] [...] │
│ Machine A Machine B etc. │
│ │
│ ring buffer → chrome.storage.local │
│ (60 readings × N machines × M metrics) │
└─────────────────────────────────────────┘

Pros

  • Zero server to run or maintain
  • Machines stay purely local
  • Works offline (cached values)
  • Config lives in existing YAML

Cons

  • Browser must be on same network
  • Extension must stay open to poll
B

Thin Hub Server

A small Node.js server (new workspace in monorepo) polls Dash./Glances, buffers history, exposes one API. Extension talks only to the hub — same network not required.

[Dash. :3001] Machine A ──pull──┐
[Glances :61208] Machine B ─pull──┤ Hub (apps/monitor)
[Glances] Machine C ────────pull──┘ └─ ring buffer (memory)
                            └─ REST API
                        Extension polls Hub

Pros

  • Works from any network / device
  • History persists even when browser closed
  • One endpoint for the extension

Cons

  • Must run & maintain a server process
  • Extra infra for a homelab tool

In both cases, the config looks the same in your YAML:

monitors:
  - name: "Mac Studio"
    type: dashdot
    url: "http://192.168.1.10:3001"
  - name: "NAS"
    type: glances
    url: "http://192.168.1.20:61208"
